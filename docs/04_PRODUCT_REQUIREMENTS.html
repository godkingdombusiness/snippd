ENVIRONMENT Variable in Edge Functions

The ENVIRONMENT secret in Supabase controls which environment label appears in security alert emails and event logs. Set this correctly per project:

# Production
supabase secrets set ENVIRONMENT=production

# Staging
supabase secrets set ENVIRONMENT=staging

Environment Variables Reference

Client App (.env.local)

These variables are bundled into the app build. EXPO_PUBLIC_ prefix means they are accessible in the JavaScript bundle. Never put service_role or secrets with this prefix.

EXPO_PUBLIC_SUPABASE_URL
Your Supabase project URL. Found in Supabase Dashboard → Settings → API
Client app
EXPO_PUBLIC_SUPABASE_ANON_KEY
Public anon key for Supabase client. Safe to bundle — RLS protects data. Never use service_role key here.
Client app
EXPO_PUBLIC_SECURITY_INGEST_PUBLIC_TOKEN
Read-only public token for client → security-ingest calls. Separate from the server-side SECURITY_INGEST_SECRET.
Client app
EXPO_PUBLIC_FIELD_ENC_KEY
AES-256-GCM field encryption key. ⚠️ Currently EXPO_PUBLIC_ — should be migrated to server-side vault.
Client app (migrate)

Supabase Edge Function Secrets (Vault)

Set via supabase secrets set KEY=value. Never committed to source code. Accessible as Deno.env.get('KEY') in Edge Functions.

SUPABASE_URL
Auto-injected by Supabase runtime. Do not set manually.
Auto-injected
SUPABASE_SERVICE_ROLE_KEY
Full DB access bypassing RLS. Only available inside Edge Functions. NEVER expose to client.
Supabase vault
SECURITY_INGEST_SECRET
Server-side secret for Edge Function authentication. Different from the public client token. 32-byte hex minimum.
Supabase vault
GEMINI_API_KEY
Google AI Studio API key for Gemini 1.5 Flash. Used only by gemini-proxy function.
Supabase vault
RESEND_API_KEY
Resend transactional email API key. Used only by security-alert and security-escalate functions.
Supabase vault
ALERT_TO_EMAIL
Security alert recipient. Default: ddavis@getsnippd.com
Supabase vault
ALERT_FROM_EMAIL
Security alert sender. Default: security@getsnippd.com
Supabase vault
ENVIRONMENT
Environment label in alerts: 'production' or 'staging'
Supabase vault

Local Development Setup

Prerequisites

  • Node.js 20+ LTS
  • npm 10+
  • Expo CLI: npm install -g expo-cli
  • Supabase CLI: npm install -g supabase
  • Expo Go app on physical iOS/Android device (for testing)
  • Android Studio or Xcode (for simulator builds)

First-Time Setup

1
Clone and install
git clone https://github.com/your-org/snippd.git
cd snippd
npm install
2
Create .env.local
Copy the example and fill in values from Supabase Dashboard → Settings → API: 
cp .env.example .env.local
# Edit .env.local with your Supabase URL and anon key
3
Run migrations in Supabase
In Supabase Dashboard → SQL Editor, run these files in order: 
sql/00_rls_policies.sql
sql/01_rate_limiting.sql
sql/02_audit_triggers.sql
sql/03_fresh_start.sql
sql/04_households.sql
sql/05_referrals.sql
sql/06_security_schema.sql
4
Set Supabase secrets
supabase login
supabase link --project-ref gsnbpfpekqqjlmkgvwvb
supabase secrets set GEMINI_API_KEY="..." RESEND_API_KEY="..." # etc.
5
Deploy Edge Functions
supabase functions deploy gemini-proxy
supabase functions deploy security-ingest
supabase functions deploy security-alert
supabase functions deploy security-escalate
6
Start the app
npm start        # Starts on port 8082
# Scan QR with Expo Go app on device, or press 'a' for Android, 'i' for iOS simulator

Migration File Inventory

FileDescriptionIdempotent?Dependencies
sql/00_rls_policies.sqlRow-Level Security policies for all core tablesYes (DROP POLICY IF EXISTS)Tables must exist

Migration Procedure (Production)

1
Backup first
Supabase Dashboard → Database → Backups → "Create Backup Now"
2
Run in Supabase SQL Editor
Copy file contents, paste into SQL Editor, run. Watch for errors before proceeding to next file.
3
Verify
Run a quick sanity check: 
SELECT table_name FROM information_schema.tables WHERE table_schema = 'public';
SELECT proname FROM pg_proc WHERE proname LIKE 'fn_%' OR proname LIKE 'close_%' OR proname LIKE 'check_%';
SELECT * FROM approved_domains;  -- should show 6 seeded domains
4
Test RLS
Attempt to read another user's row with their JWT — should be blocked. Attempt to insert into system_audit_logs — should be blocked.

Adding New Migrations

  • Name files sequentially: 07_feature_name.sql
  • Always use IF NOT EXISTS for CREATE TABLE/INDEX
  • Always use CREATE OR REPLACE for functions
  • Test on staging before running on production
  • Document the migration purpose in a comment at the top of the file

Scheduled Jobs

security-escalate Cron Job

The security-escalate Edge Function must run every 5 minutes. Set up via pg_cron in Supabase SQL Editor:

-- Enable pg_cron extension (if not already enabled)
CREATE EXTENSION IF NOT EXISTS pg_cron;

-- Create the cron job
SELECT cron.schedule(
  'security-escalate-job',
  '*/5 * * * *',
  $$
    SELECT net.http_post(
      url        := current_setting('app.supabase_url') || '/functions/v1/security-escalate',
      headers    := jsonb_build_object(
        'Content-Type',     'application/json',
        'x-security-token', current_setting('app.security_ingest_secret')
      ),
      body       := '{}'::jsonb
    );
  $$
);

-- Set app-level settings used by cron
ALTER DATABASE postgres SET app.supabase_url = 'https://gsnbpfpekqqjlmkgvwvb.supabase.co';
ALTER DATABASE postgres SET app.security_ingest_secret = '<your-secret>';

-- Verify job is scheduled
SELECT * FROM cron.job WHERE jobname = 'security-escalate-job';

Security Event Data Cleanup Cron

-- Purge resolved LOW severity events older than 30 days
SELECT cron.schedule(
  'purge-low-security-events',
  '0 3 * * *',  -- Daily at 3 AM
  $$ SELECT fn_purge_low_severity_events(); $$
);

Cron Job Monitoring

-- Check recent cron job runs
SELECT * FROM cron.job_run_details
WHERE jobid = (SELECT jobid FROM cron.job WHERE jobname = 'security-escalate-job')
ORDER BY start_time DESC LIMIT 10;

-- If cron fails for 30+ min, security-escalate health check will email you

Rollback Procedures

Edge Function Rollback

Supabase does not provide one-click rollback for Edge Functions. Rollback by redeploying the previous version from git:

git checkout <previous-commit-hash> -- supabase/functions/security-ingest/index.ts
supabase functions deploy security-ingest

Database Migration Rollback

Migrations are additive (no DROP statements in normal workflow). For emergency rollback:

-- 1. Restore from backup in Supabase Dashboard → Database → Backups
-- 2. Or manually revert specific changes (e.g., drop new columns)
-- Always test rollback in staging first

Mobile App Rollback (OTA)

# Roll back to previous OTA channel
eas update --branch production --message "Rollback: revert to previous build"
# Native (store build) rollbacks require submitting a previous build to the stores
