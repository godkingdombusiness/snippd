Threat Model

The Snippd threat model follows the STRIDE framework applied to a mobile grocery app handling user financial data (budgets, spending, payment-adjacent credits), personal data (dietary preferences, household composition), and authentication credentials.

Security ingest secretHIGHAbility to inject false security events, suppress real ones
Gemini API keyHIGHFinancial abuse (API cost), data leakage via model inputs
Resend API keyHIGHEmail impersonation, phishing from getsnippd.com domain
User financial data (budgets, spending)HIGHPrivacy violation, regulatory exposure
User PII (name, email, household)HIGHPrivacy violation, CCPA/GDPR liability

Threat Actors

External Attacker
Unauthenticated actor targeting login endpoints, attempting credential stuffing, injection attacks, or API abuse to extract data or earn fraudulent credits.
Malicious Insider / Compromised Account
Authenticated user attempting IDOR to access other households' data, escalate privileges, or abuse the referral/coupon system.
Automated Bot / Scraper
Automated systems targeting the deal feed, creating fake accounts for referral fraud, or scraping product/pricing data.
Developer Workflow Attack
Supply chain attacks, malicious package releases, prompt injection via AI tools, or repo config bypass targeting developer machines with prod credentials.

Section 3

Authentication & Session Security

Authentication Flow

  1. Sign Up: Email + password → Supabase Auth creates auth.users row → trigger creates profiles row → email verification sent
  2. Sign In: Email + password → Supabase PKCE flow → JWT issued → stored in SecureStore
  3. MFA Challenge: If mfa_enabled=true on profile → navigate to MFAVerifyScreen → TOTP code entry → supabase.auth.mfa.challengeAndVerify()
  4. Session Persistence: autoRefreshToken: true silently refreshes JWTs before expiry
  5. Inactivity Logout: sessionGuard.js monitors touch events via PanResponder; resets timer on any touch; after 30 minutes of inactivity → supabase.auth.signOut()

Brute Force Detection

Client-side detection in SecurityMonitor.trackLoginFailure():

  • In-memory map keyed by ip:route
  • First failure → fires LOGIN_FAILED event
  • 5+ failures within 5-minute window → fires BRUTE_FORCE_IN_PROGRESS (MEDIUM severity)
  • Successful login after 3+ failures → fires BRUTE_FORCE_SUCCESS (HIGH severity) → email alert
  • Window resets after 5 minutes of inactivity
Limitation: Client-side brute force tracking is per-session (in-memory). A server-side check using the api_rate_limit_log table should also be used to detect cross-session or distributed brute force attacks.

Data Security & Encryption

Field-Level Encryption

Sensitive fields are encrypted client-side using AES-256-GCM before being stored in Supabase. The encryption key is currently sourced from EXPO_PUBLIC_FIELD_ENC_KEY.

// lib/fieldEncryption.js
encryptField(plaintext: string) → { ciphertext, iv, tag }
decryptField({ ciphertext, iv, tag }) → plaintext

Algorithm : AES-256-GCM
Key source : process.env.EXPO_PUBLIC_FIELD_ENC_KEY (32 bytes hex)
IV         : 12 bytes random per encryption (stored with ciphertext)
Tag        : 16 bytes GCM auth tag (tamper detection)
CRITICAL Gap: The encryption key is in EXPO_PUBLIC_ prefix, meaning it's bundled into the client app binary and extractable via reverse engineering. Production requires the key to live in Supabase vault and encryption/decryption to occur server-side or via a key-derivation service.

Section 5

Security Monitoring Pipeline

Pipeline Architecture

App (SecurityMonitor.js)
    │
    │  POST /security-ingest
    │  x-security-token: PUBLIC_TOKEN
    ▼
[security-ingest Edge Function]
    ├── 1. Auth: verify x-security-token
    ├── 2. Sanitize: strip non-ASCII, truncate
    ├── 3. Fingerprint: SHA-256(event_type|user_id|ip|route|window_bucket)
    ├── 4. Dedup: count_events_in_window() RPC
    │       └── if LOW/MEDIUM duplicate in window → return 'deduplicated'
    ├── 5. IP Block check: is_blocked() RPC
    │       └── if blocked → return 'blocked'
    ├── 6. Score severity: CRITICAL_TYPES → 95, HIGH → 75, MEDIUM → 45, LOW → 5
    │       └── Repeat escalation: MEDIUM+10 → HIGH, HIGH+5 → CRITICAL
    ├── 7. INSERT security_events
    ├── 8. if CRITICAL + ip → upsert blocked_indicators (24h)
    └── 9. if alert_required → async POST /security-alert

[security-alert Edge Function]
    ├── 1. Fetch full event from security_events
    ├── 2. Cooldown check (CRITICAL=60s, HIGH=300s, MEDIUM=3600s)
    ├── 3. Build email (HTML + text)
    ├── 4. Create security_alerts record (PENDING)
    ├── 5. POST Resend API → ddavis@getsnippd.com
    └── 6. Update alert status (SENT/FAILED) + log to alert_deliveries

[security-escalate Cron — every 5 min]
    ├── 1. Retry FAILED alerts (exp. backoff, max 3)
    ├── 2. Volume escalation: MEDIUM ≥10/hr → inject ESCALATED_VOLUME_PATTERN HIGH
    ├── 3. Health check: if 0 events in 30 min → direct Resend alert (pipeline bypass)
    └── 4. Expire stale blocked_indicators (active=false)

Alert Cooldown Windows

SeverityCooldownReasoning
CRITICAL60 secondsImmediate repeated alerting — operator must act now
HIGH5 minutesUrgent but allows burst deduplication
MEDIUM1 hourDigest-level — avoid alert fatigue
LOW24 hours (no email)Suppressed — log only

Pipeline Health Monitoring

The security-escalate job includes a dead-man's switch: if no security events have been recorded in 30 minutes, it sends a CRITICAL health failure email directly via Resend (bypassing the normal pipeline, which may itself be broken). This ensures monitoring tampering is detected even if the ingest function is down.

Section 7

Incident Response Playbooks

BRUTE_FORCE_SUCCESS

Playbook: Brute Force Success

  1. Receive HIGH alert email → confirm event in security_events table
  2. Immediately invalidate all sessions for the affected user: Supabase Dashboard → Auth → Users → Sign Out All Devices
  3. Temporarily disable account (set profiles.is_blocked=true if field exists, or use Supabase Auth admin ban)
  4. Block the source IP: INSERT INTO blocked_indicators (indicator_type, indicator_value, reason, severity, expires_at) VALUES ('IP', '<ip>', 'Brute force success', 'HIGH', now()+interval '7 days')
  5. Email affected user: "We detected unusual sign-in activity. Your account has been temporarily secured."
  6. Review all actions taken by this account in the last 24h via system_audit_logs
  7. If financial data accessed: document scope and assess notification requirements
  8. Restore account access with forced password reset + MFA enrollment
SERVICE_ROLE_KEY_EXPOSED / SECRET_EXFILTRATION_ATTEMPT

Playbook: Secret/Credential Exposure — EMERGENCY

  1. STOP all deployments immediately
  2. Rotate Supabase service_role key: Dashboard → Settings → API → Rotate Keys
  3. Update all Supabase Edge Function secrets with new key: supabase secrets set SUPABASE_SERVICE_ROLE_KEY=<new>
  4. Rotate all other potentially exposed secrets (Gemini, Resend, SECURITY_INGEST_SECRET)
  5. Check git history for committed secrets: git log --all -S "service_role"
  6. If committed: use git-filter-repo to purge, force-push, invalidate all forks
  7. Review audit logs for any unauthorized DB access during exposure window
  8. Document exposure window, actions taken, data accessed → for compliance reporting
ADMIN_ACCOUNT_TAKEOVER

Playbook: Admin Account Takeover — EMERGENCY

  1. Immediately revoke all sessions for the admin account
  2. Disable admin account in Supabase Auth
  3. Rotate SECURITY_INGEST_SECRET (attacker may have visibility into monitoring)
  4. Review all changes made by the admin account in last 72h via system_audit_logs
  5. Check if any new admin accounts were created or privileges escalated
  6. Verify security monitoring pipeline integrity (check escalate job is still running)
  7. Re-enable admin access only after forensic review and MFA re-enrollment
MONITORING_TAMPERING

Playbook: Security Monitoring Tampering — EMERGENCY

  1. Treat as highest-priority incident — attacker may be actively covering tracks
  2. Verify security-ingest edge function is running: test with manual POST
  3. Check pg_cron job for security-escalate: SELECT * FROM cron.job WHERE jobname LIKE 'security%'
  4. Review Supabase Edge Function logs for unauthorized restarts or configuration changes
  5. Verify SECURITY_INGEST_SECRET has not been changed
  6. Check blocked_indicators table for unauthorized entries or mass deactivation
  7. Re-deploy all security Edge Functions from known-good source
  8. Escalate to incident response team — assume breach until proven otherwise
REFERRAL_FRAUD_ATTEMPT

Playbook: Referral / Credit Fraud

  1. Identify affected referrals: SELECT * FROM referrals WHERE fraud_score > 50
  2. Block all pending credit releases for flagged referrals: set referrer_credits_status='blocked'
  3. Review account creation patterns for bulk signups from same IP/device
  4. Check for identical dietary preferences or device fingerprints across accounts
  5. Permanently ban confirmed fraud accounts and reclaim awarded credits
  6. Increase fraud_score threshold if pattern is novel

Security Testing Plan

1. Automated — Pre-Commit

  • detect-secrets scan on all staged files
  • ESLint security plugin for common patterns (eval, innerHTML, etc.)
  • npm audit for known CVEs in dependencies

2. Authentication Testing

  • Verify password reset token single-use
  • Test session invalidation on signOut across devices
  • Verify MFA cannot be bypassed by direct API call
  • Test inactivity logout fires at exactly 30 minutes
  • Attempt TOTP replay attack (same code used twice)

3. Authorization / RLS Testing

  • Attempt to query another user's profile row with own JWT
  • Attempt to modify household_members with SHOPPER role
  • Attempt to read security_events with non-admin JWT
  • Attempt to insert into system_audit_logs directly
  • Verify RLS blocks service_role-only operations when using anon key

4. Injection Testing (security-ingest validation)

  • Send SQL injection payload in summary field — verify sanitise() strips it
  • Send XSS payload in route field — verify stripped to ASCII only
  • Send excessively long strings — verify truncation at maxLen
  • Send non-UTF8 bytes — verify stripped by replace(/[^\x20-\x7E\n\r\t]/g, '')

5. Security Monitoring Pipeline Testing

  • Fire a test LOGIN_FAILED event → verify appears in security_events
  • Fire a test BRUTE_FORCE_SUCCESS event → verify email received within 60 seconds
  • Fire CRITICAL event → verify IP is added to blocked_indicators
  • Fire same event twice within window → verify second is deduplicated
  • Stop firing events for 31 minutes → verify health check alert fires
