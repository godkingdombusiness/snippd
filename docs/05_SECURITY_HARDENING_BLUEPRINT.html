Part 1

Executive Security Architecture

Snippd's security model is built on a single non-negotiable premise: the mobile client is fully untrusted. Every security decision flows from this. A user who roots their device, proxies network traffic, or reverse-engineers the app binary must never be able to manipulate balances, escalate privileges, claim rewards, or access other users' data — no matter what they send to the server.

Component Trust Levels

React Native Client
UNTRUSTED
Runs on user-controlled hardware. Binary extractable. Can be modified, proxied, replayed. Zero trust — all claims validated server-side.
Supabase Anon Key
UNTRUSTED
Public. Anyone who downloads the app has it. Only grants access as restricted by RLS. Never grants privilege on its own.
Supabase Auth JWT
SEMI-TRUSTED
Cryptographically signed by Supabase. Proves identity of auth.uid(). Cannot prove intent, honesty of payload, or absence of compromise.
Edge Functions (public routes)
SEMI-TRUSTED
Server-controlled code but invocable by any authenticated client. Must apply own rate limiting, validation, and business rule enforcement.
Edge Functions (internal, HMAC)
TRUSTED
Function-to-function calls signed with shared HMAC secret. Not publicly invocable. Verified on receipt.
PostgreSQL + RLS
TRUSTED
Final enforcement layer. RLS runs with every query regardless of caller. Cannot be bypassed by client.
Supabase Service Role
TRUSTED (controlled)
Bypasses RLS. Must exist only in Edge Function vault secrets. Never in client bundle. Rotate on any suspected exposure.
Developer Machine
SEMI-TRUSTED
May contain credentials, may be compromised by malicious packages, AI tooling, or supply chain attacks. Must be isolated from production.
pg_cron Jobs
SEMI-TRUSTED
Runs as postgres superuser role by default. Jobs must be restricted, logged, and monitored. Cannot be created/modified by unprivileged roles.

Part 3

Critical Vulnerabilities — Eliminate Immediately

⚠ CRITICAL-1: EXPO_PUBLIC_FIELD_ENC_KEY — Encryption Key in Client Bundle

Why it's dangerous: Any value with the EXPO_PUBLIC_ prefix is bundled into the JavaScript binary shipped to the App Store. Anyone who downloads the app, extracts the JS bundle (trivial on Android; possible on iOS with jailbreak), or uses a decompiler can read this value. Your AES-256-GCM "field encryption" provides zero security because the key is public. An attacker who obtains it can decrypt all encrypted fields at rest in your database.

Fix: Treat this key as fully compromised. Rotate it immediately. Move all encryption/decryption to a server-side Edge Function. The client sends plaintext to an authenticated Edge Function which encrypts server-side using a key stored only in Supabase vault. See Part 10 for full redesign.

⚠ CRITICAL-2: Local Development Against Production Supabase Project

Why it's dangerous: A developer running npm start with the production .env connects to live user data. A mistake (wrong query, accidental delete, test data mutation) affects real users. Worse: if the developer's machine is compromised, the attacker has access to production credentials and can read/write all data.

Fix: Create a separate Supabase project for development. Developer .env.local must point to the dev project. Production credentials must never appear in any local file — they are injected only by CI/CD secrets at deploy time. See Part 11.

⚠ CRITICAL-3: Client-Side Credit / Reward Logic

Why it's dangerous: Any credit or reward logic that executes on the client (or trusts client-provided amounts/types) can be manipulated. A proxying user can send arbitrary amounts to a credits endpoint. SecurityMonitor running client-side is easy to bypass — just don't call it.

Fix: All credit issuance, referral verification, and reward calculation must run exclusively in SECURITY DEFINER Postgres functions or Edge Functions using the service role. Client sends a request (e.g., "complete trip"). Server validates, calculates, and writes to append-only credit_ledger. Client never sends an amount.

⚠ CRITICAL-4: Client-Side Brute Force Detection Only

Why it's dangerous: trackLoginFailure() in SecurityMonitor.js uses in-memory state per session. An attacker who makes login requests directly to Supabase Auth (bypassing the app entirely) generates zero client-side events. The detection is completely invisible to direct API attacks.

Fix: Implement server-side rate limiting in a login Edge Function that proxies all auth attempts. Track failures in api_rate_limit_log by IP. After 5 failures in 5 minutes, return 429 and fire security event server-side. Client-side tracking is supplemental, never authoritative.

⚠ CRITICAL-5: Missing Replay Protection on Events and Rewards

Why it's dangerous: If a "complete trip" or "claim reward" request can be replayed, an attacker can capture one successful request and resend it repeatedly to earn unlimited credits. No idempotency key means every retry is a potential double-award.

Fix: Every state-changing operation must include an idempotency_key (UUID v4, client-generated) and a request_timestamp. The server rejects requests older than ±5 minutes. Idempotency keys are stored in processed_requests table with UNIQUE constraint. Duplicate key = return cached response. See Part 8.

⚠ HIGH-6: Missing Step-Up Authentication for Sensitive Actions

Why it's dangerous: Actions like disabling MFA, changing email/password, or changing a household role can be taken by anyone with a valid session — even a stolen or idle token. A 30-minute session window is a large attack surface for a session hijacker.

Fix: Require re-authentication (password re-entry + TOTP) before sensitive actions. Issue a short-lived elevated_session_token (15-minute TTL) stored in SecureStore. All sensitive Edge Functions validate this token. See Part 9.

⚠ HIGH-7: No Emergency Kill Switch / Feature Flag System

Why it's dangerous: If a referral fraud outbreak is detected, there is currently no way to disable the referral system without a full code deploy. Minutes of delay during an active fraud attack can cost thousands of credits.

Fix: Implement a feature_flags table in Supabase. Every critical subsystem checks its flag before executing. Flags can be toggled instantly via SQL with zero code deploy. See Part 14.

Part 4

Secure Backend Enforcement Model

The Three-Layer Rule

Client can request. Server decides. Database enforces.

The client is a UI that signals intent. It never carries authority. Every privileged decision — who gets credits, what role someone has, whether a referral is valid — is made by Edge Functions or Postgres functions using the service role and verified against database state. RLS is the last line of enforcement that runs for every query regardless of caller.

Never Trust These Client Claims

Values the Backend Must Never Accept at Face Value from the Client
  • stash_credits — any amount or delta. Balance is calculated from credit_ledger SUM server-side.
  • subscription_type — 'monthly', 'lifetime', or any plan string. Verified against IAP receipt server-side.
  • role — 'STACK_MANAGER', 'SHOPPER', 'VIEWER'. Role is read from household_members by user_id, never accepted as input.
  • referral_code (for credit claims) — must be validated against database, not trusted as-is.
  • savings_cents — any claimed savings amount. Server recalculates from actual receipt data.
  • trip_total_cents — claimed trip total. Verified against OCR-extracted items stored server-side.
  • fraud_score — any fraud risk level. Calculated by server, not accepted from client.
  • is_admin, is_security_admin — any privilege flag. Checked in DB via helper functions only.
  • user_id in any body payload — always use auth.uid() from verified JWT, never body.user_id.
  • household_id without membership verification — always validate the caller is a member of that household.
  • elevated_session, is_elevated — any self-asserted step-up status. Validate elevated_session_token from vault.
  • created_at, updated_at timestamps — always generated by the server (NOW()).
  • event severity in security events — calculated by server severity engine, never accepted from client payload.
Part 5

Supabase RLS Hardening Plan

Core Policy Model: Deny-by-Default

Every table has RLS enabled. No policy = no access. Every policy is additive (PERMISSIVE) and must be explicitly proven necessary. When in doubt, deny.

RLS Helper Functions

-- ============================================================
-- HELPER FUNCTIONS — install before any RLS policies
-- ============================================================

-- Returns true if current user is a member of the given household
CREATE OR REPLACE FUNCTION is_household_member(p_household_id UUID)
RETURNS BOOLEAN LANGUAGE sql SECURITY DEFINER STABLE AS $$
  SELECT EXISTS(
    SELECT 1 FROM household_members
    WHERE household_id = p_household_id
      AND user_id = auth.uid()
  );
$$;

-- Returns current user's role in a household, or NULL if not a member
CREATE OR REPLACE FUNCTION household_role(p_household_id UUID)
RETURNS TEXT LANGUAGE sql SECURITY DEFINER STABLE AS $$
  SELECT role FROM household_members
  WHERE household_id = p_household_id AND user_id = auth.uid();
$$;

-- Returns true if current user is a security admin
CREATE OR REPLACE FUNCTION is_security_admin()
RETURNS BOOLEAN LANGUAGE sql SECURITY DEFINER STABLE AS $$
  SELECT EXISTS(
    SELECT 1 FROM profiles
    WHERE id = auth.uid() AND is_admin = TRUE
  );
$$;

-- Returns true if a flag is enabled (kill switch check)
CREATE OR REPLACE FUNCTION is_feature_enabled(p_flag TEXT)
RETURNS BOOLEAN LANGUAGE sql SECURITY DEFINER STABLE AS $$
  SELECT COALESCE(
    (SELECT enabled FROM feature_flags WHERE flag_name = p_flag),
    FALSE
  );
$$;

Per-Table RLS Policies

-- ============================================================
-- PROFILES — users read/update own row only
-- Server strips privileged fields before any client update
-- ============================================================
ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;

CREATE POLICY "profiles_select_own" ON profiles
  FOR SELECT USING (auth.uid() = id);

CREATE POLICY "profiles_update_own" ON profiles
  FOR UPDATE USING (auth.uid() = id)
  WITH CHECK (auth.uid() = id);

-- ❌ NEVER: no INSERT policy for client (service role only via trigger)
-- ❌ NEVER: no DELETE policy (account deletion goes through Edge Function)

-- ============================================================
-- HOUSEHOLDS — members read; STACK_MANAGER updates
-- ============================================================
ALTER TABLE households ENABLE ROW LEVEL SECURITY;

CREATE POLICY "households_select_members" ON households
  FOR SELECT USING (is_household_member(id));

CREATE POLICY "households_update_manager" ON households
  FOR UPDATE USING (household_role(id) = 'STACK_MANAGER')
  WITH CHECK (household_role(id) = 'STACK_MANAGER');

-- ============================================================
-- HOUSEHOLD_MEMBERS — members read own household's rows
-- NO client INSERT/UPDATE/DELETE — all via fn_add_member() etc.
-- ============================================================
ALTER TABLE household_members ENABLE ROW LEVEL SECURITY;

CREATE POLICY "hm_select_own_household" ON household_members
  FOR SELECT USING (is_household_member(household_id));

-- ============================================================
-- CREDIT_LEDGER — users read own; no client INSERT (server only)
-- ============================================================
ALTER TABLE credit_ledger ENABLE ROW LEVEL SECURITY;

CREATE POLICY "ledger_select_own" ON credit_ledger
  FOR SELECT USING (user_id = auth.uid());

-- No INSERT/UPDATE/DELETE policies — service role only

-- ============================================================
-- HOUSEHOLD_CART_ITEMS — scoped to household membership
-- ============================================================
ALTER TABLE household_cart_items ENABLE ROW LEVEL SECURITY;

CREATE POLICY "cart_select_household" ON household_cart_items
  FOR SELECT USING (is_household_member(household_id));

CREATE POLICY "cart_insert_shopper+" ON household_cart_items

    is_household_member(household_id)

-- ============================================================

-- SYSTEM_AUDIT_LOGS — no client access whatsoever

Shared Edge Function Security Middleware

// supabase/functions/_shared/middleware.ts

export interface RequestContext {
  userId: string
  jwt: string
  correlationId: string
  isElevated: boolean
}

// ── JWT Verification ──────────────────────────────────────────────────────
export async function requireAuth(req: Request, supabase: SupabaseClient): Promise<RequestContext> {
  const auth = req.headers.get('Authorization')
  if (!auth?.startsWith('Bearer ')) throw unauthorizedError('Missing JWT')

  const jwt = auth.slice(7)
  const { data: { user }, error } = await supabase.auth.getUser(jwt)
  if (error || !user) throw unauthorizedError('Invalid JWT')

  const correlationId = crypto.randomUUID()
  return { userId: user.id, jwt, correlationId, isElevated: false }
}

// ── Step-Up Token Validation ──────────────────────────────────────────────
export async function requireStepUp(req: Request, ctx: RequestContext, supabase: SupabaseClient): Promise<void> {
  const token = req.headers.get('x-step-up-token')
  if (!token) throw forbiddenError('Step-up authentication required')

  const { data, error } = await supabase
    .from('elevated_sessions')
    .select('expires_at, user_id, used')
    .eq('token', token)
    .eq('user_id', ctx.userId)
    .single()

  if (error || !data) throw forbiddenError('Invalid step-up token')
  if (data.used) throw forbiddenError('Step-up token already used')
  if (new Date(data.expires_at) < new Date()) throw forbiddenError('Step-up token expired')

  ctx.isElevated = true
}

// ── Rate Limiting ─────────────────────────────────────────────────────────
export async function checkRateLimit(
  supabase: SupabaseClient, userId: string, ip: string,
  endpoint: string, maxPerUser: number, maxPerIp: number, windowSecs: number
): Promise<void> {
  const [userCheck, ipCheck] = await Promise.all([
    supabase.rpc('check_rate_limit', { p_key: userId, p_endpoint: endpoint, p_max: maxPerUser, p_window: windowSecs }),
    supabase.rpc('check_rate_limit', { p_key: ip, p_endpoint: `ip:${endpoint}`, p_max: maxPerIp, p_window: windowSecs })
  ])
  if (userCheck.data === false) throw rateLimitError('User rate limit exceeded')
  if (ipCheck.data === false) throw rateLimitError('IP rate limit exceeded')
}

// ── Timestamp / Replay Guard ──────────────────────────────────────────────
export function validateTimestamp(requestTimestamp: string, maxSkewMs = 300000): void {
  const ts = new Date(requestTimestamp).getTime()
  const now = Date.now()
  if (Math.abs(now - ts) > maxSkewMs) throw badRequestError('Request timestamp out of allowed window (±5 min)')
}

// ── HMAC Signature Verification (internal function-to-function) ───────────
export async function verifyHmac(req: Request, secret: string): Promise<void> {
  const sig = req.headers.get('x-ingest-sig')
  const ts  = req.headers.get('x-timestamp')
  const body = await req.text()

  const key = await crypto.subtle.importKey(
    'raw', new TextEncoder().encode(secret), { name: 'HMAC', hash: 'SHA-256' }, false, ['sign', 'verify']
  )
  const expected = await crypto.subtle.sign('HMAC', key, new TextEncoder().encode(`${ts}.${body}`))
  const expectedHex = [...new Uint8Array(expected)].map(b => b.toString(16).padStart(2, '0')).join('')

  if (sig !== expectedHex) throw unauthorizedError('Invalid HMAC signature')
  validateTimestamp(ts!) // also check timestamp freshness
}

// ── Safe Outbound Fetch (domain allowlist) ────────────────────────────────
export async function guardedFetch(url: string, opts: RequestInit, supabase: SupabaseClient): Promise<Response> {
  const domain = new URL(url).hostname
  const { data } = await supabase.from('approved_domains').select('domain').eq('domain', domain).eq('active', true).maybeSingle()
  if (!data) throw new Error(`Blocked outbound to unapproved domain: ${domain}`)
  return fetch(url, opts)
}
Part 7

Rewards / Referral / Credits Anti-Fraud Design

Append-Only Credit Ledger Schema

-- ============================================================
-- CREDIT_LEDGER — every credit event is a row, never an UPDATE
-- Balance = SUM(amount) for user. Always server-written.
-- ============================================================
CREATE TABLE credit_ledger (
  id              UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID        NOT NULL REFERENCES profiles(id),
  amount          INTEGER     NOT NULL,        -- positive = credit, negative = debit
  transaction_type TEXT       NOT NULL,        -- WEEKLY_TRIP|REFERRAL_INSTANT|REFERRAL_RELEASE|PROMO|LEVEL_BONUS|ADMIN_ADJUST|DEBIT
  reference_id    UUID,                        -- trip_id, referral_id, promo_id, etc.
  idempotency_key TEXT        UNIQUE NOT NULL, -- prevents double-award
  issued_by       TEXT        NOT NULL,        -- 'system'|'admin:{admin_id}'
  metadata        JSONB       DEFAULT '{}',
  created_at      TIMESTAMPTZ DEFAULT NOW()
);

-- SECURITY DEFINER: only callable with elevated permissions
-- Client never calls this directly — only Edge Functions with service role
CREATE OR REPLACE FUNCTION issue_credits(
  p_user_id        UUID,
  p_amount         INTEGER,
  p_type           TEXT,
  p_reference_id   UUID,
  p_idempotency    TEXT,
  p_issued_by      TEXT
) RETURNS credit_ledger
LANGUAGE plpgsql SECURITY DEFINER AS $$
DECLARE
  v_row credit_ledger;
  v_blocked BOOLEAN;
  v_flag_count INTEGER;
BEGIN
  -- 1. Check if user is blocked for fraud
  SELECT EXISTS(SELECT 1 FROM fraud_flags WHERE user_id = p_user_id AND auto_blocked = TRUE) INTO v_blocked;
  IF v_blocked THEN
    RAISE EXCEPTION 'User is blocked from receiving credits due to fraud flag';
  END IF;

  -- 2. Check kill switch
  IF NOT is_feature_enabled('rewards_enabled') THEN
    RAISE EXCEPTION 'Rewards subsystem is currently disabled';
  END IF;

  -- 3. Insert with idempotency (ON CONFLICT DO NOTHING returns existing row)
  INSERT INTO credit_ledger(user_id, amount, transaction_type, reference_id, idempotency_key, issued_by)
  VALUES(p_user_id, p_amount, p_type, p_reference_id, p_idempotency, p_issued_by)
  ON CONFLICT(idempotency_key) DO NOTHING
  RETURNING * INTO v_row;

  -- 4. Anomaly: flag if user has received >200 credits in last 24h
  SELECT COUNT(*) INTO v_flag_count
  FROM credit_ledger
  WHERE user_id = p_user_id AND amount > 0
    AND created_at > NOW() - '24h'::INTERVAL;

  IF v_flag_count > 200 THEN
    INSERT INTO fraud_flags(user_id, flag_type, risk_score, evidence)
    VALUES(p_user_id, 'CREDIT_ABUSE', 80, jsonb_build_object('credits_24h', v_flag_count));
  END IF;

  RETURN v_row;
END;
$$;

Idempotency Enforcement — TypeScript

// supabase/functions/_shared/idempotency.ts

export async function withIdempotency<T>(
  supabaseAdmin: SupabaseClient,
  idempotencyKey: string,
  endpoint: string,
  userId: string,
  requestBody: unknown,
  handler: () => Promise<T>
): Promise<{ result: T; cached: boolean }> {

  // 1. Validate idempotency key format
  if (!/^[0-9a-f-]{36}$/.test(idempotencyKey)) throw badRequestError('Invalid idempotency_key format')

  // 2. Check if already processed
  const { data: existing } = await supabaseAdmin
    .from('processed_requests')
    .select('response_body, response_status, expires_at')
    .eq('idempotency_key', idempotencyKey)
    .eq('user_id', userId)
    .maybeSingle()

  if (existing && new Date(existing.expires_at) > new Date()) {
    return { result: existing.response_body as T, cached: true }
  }

  // 3. Compute request hash for dedup within same key but different body
  const bodyText = JSON.stringify(requestBody)
  const hashBytes = await crypto.subtle.digest('SHA-256', new TextEncoder().encode(`${endpoint}:${bodyText}`))
  const requestHash = [...new Uint8Array(hashBytes)].map(b => b.toString(16).padStart(2, '0')).join('')

  // 4. Execute handler
  const result = await handler()

  // 5. Record as processed
  await supabaseAdmin.from('processed_requests').insert({
    idempotency_key: idempotencyKey,
    endpoint,
    user_id: userId,
    request_hash: requestHash,
    response_status: 200,
    response_body: result
  })

  return { result, cached: false }
}

const MAX_SKEW_MS = 5 * 60 * 1000 // 5 minutes

function validateRequestTimestamp(ts: string): void {
  const requestTime = new Date(ts).getTime()
  if (isNaN(requestTime)) throw badRequestError('Invalid request_timestamp')
  const skew = Math.abs(Date.now() - requestTime)
  if (skew > MAX_SKEW_MS) throw badRequestError(`Request too old or future-dated. Skew: ${skew}ms`)

// Edge Function: POST /request-step-up
// Called by client BEFORE performing sensitive action

export default async function handler(req: Request) {
  const ctx = await requireAuth(req, publicClient)
  const { password, totp_code, action } = await req.json()

  // 1. Re-verify password via Supabase Auth sign-in
  const { error: authError } = await publicClient.auth.signInWithPassword({
    email: ctx.userEmail, password
  })
  if (authError) throw forbiddenError('Password verification failed')

  // 2. Verify TOTP if action requires it
  if (requiresTotp(action)) {
    const factors = await publicClient.auth.mfa.listFactors()
    const totp = factors.data?.totp?.[0]
    if (!totp) throw forbiddenError('MFA required for this action but not enrolled')
    const { error } = await publicClient.auth.mfa.challengeAndVerify({ factorId: totp.id, code: totp_code })
    if (error) throw forbiddenError('TOTP verification failed')
  }

  // 3. Issue elevated session token
  const ttlMinutes = getActionTtl(action) // 5 or 15 min based on action
  const { data: session } = await adminClient.from('elevated_sessions').insert({
    user_id: ctx.userId,
    action,
    expires_at: new Date(Date.now() + ttlMinutes * 60000).toISOString()
  }).select('token').single()

  // 4. Log the step-up event
  await logSecurityEvent('STEP_UP_AUTH_GRANTED', 'AUTH', 'MEDIUM', ctx.userId, { action })

  return Response.json({ elevated_session_token: session.token, expires_in_minutes: ttlMinutes })
}

Client-Side Flow

  1. User triggers sensitive action (e.g., "Disable MFA")
  2. App prompts: enter password + TOTP code
  3. App calls POST /request-step-up → receives elevated_session_token
  4. Token stored in memory only (not SecureStore — expires quickly)
  5. Sensitive action request includes x-step-up-token: <token> header
  6. Edge Function validates token via requireStepUp() + marks used=true
  7. Token is single-use — cannot be replayed

// supabase/functions/encrypt-field/index.ts
// ALL encryption/decryption happens here — key never leaves vault

// Key Hierarchy:
// Master Key (in Supabase vault) → Per-user Data Encryption Key (DEK)
// DEK stored encrypted in profiles.encrypted_dek
// Actual field encrypted with DEK
// This allows per-user key rotation without re-encrypting all data

async function getOrCreateUserDek(userId: string, adminClient: SupabaseClient): Promise<CryptoKey> {
  const masterKeyHex = Deno.env.get('MASTER_ENC_KEY')! // from Supabase vault only
  const masterKey = await importKey(masterKeyHex)

  const { data: profile } = await adminClient.from('profiles').select('encrypted_dek').eq('id', userId).single()

  if (profile?.encrypted_dek) {
    // Decrypt the per-user DEK using the master key
    return decryptKey(profile.encrypted_dek, masterKey)
  }

  // Generate new per-user DEK and encrypt it with master key
  const dek = await crypto.subtle.generateKey({ name: 'AES-GCM', length: 256 }, true, ['encrypt', 'decrypt'])
  const encryptedDek = await encryptKey(dek, masterKey)
  await adminClient.from('profiles').update({ encrypted_dek: encryptedDek }).eq('id', userId)
  return dek
}

export async function encryptField(userId: string, plaintext: string, adminClient: SupabaseClient): Promise<string> {
  const dek = await getOrCreateUserDek(userId, adminClient)
  const iv = crypto.getRandomValues(new Uint8Array(12))
  const ciphertext = await crypto.subtle.encrypt({ name: 'AES-GCM', iv }, dek, new TextEncoder().encode(plaintext))
  // Return as base64(iv) + '.' + base64(ciphertext)
  return `${btoa(String.fromCharCode(...iv))}.${btoa(String.fromCharCode(...new Uint8Array(ciphertext)))}`
}

Key Rotation Process

  1. Generate new MASTER_ENC_KEY: node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
  2. Set in Supabase vault: supabase secrets set MASTER_ENC_KEY=<new> MASTER_ENC_KEY_LEGACY=<old>
  3. Run migration Edge Function: re-encrypt all encrypted_dek rows using old key → new key
  4. Remove MASTER_ENC_KEY_LEGACY from vault after migration verified complete
  5. Rotate on any suspected compromise — do not wait for scheduled rotation

Incident Response Playbooks

CRITICAL — REFERRAL FRAUD OUTBREAK

Referral Farm Detected

  1. Kill switch: UPDATE feature_flags SET enabled=FALSE WHERE flag_name='referrals_enabled'
  2. Identify cluster: SELECT * FROM referral_verifications WHERE fraud_score > 70 ORDER BY created_at DESC
  3. Block accounts: INSERT INTO fraud_flags(user_id, flag_type, risk_score, auto_blocked) SELECT referee_id, 'REFERRAL_FARM', 90, TRUE FROM referrals WHERE fraud_score > 70
  4. Reverse fraudulent credits: Admin-issued negative credit_ledger entries for each flagged user
  5. Document scope in investigations table
  6. Re-enable referrals only after root cause fixed and validation tightened
CRITICAL — SERVICE_ROLE KEY EXPOSED

Credential Breach — Emergency

  1. STOP all deployments and CI/CD pipelines immediately
  2. Rotate in Supabase Dashboard: Settings → API → Rotate Keys (old key invalidated in 24h)
  3. Update all Edge Function secrets: supabase secrets set SUPABASE_SERVICE_ROLE_KEY=<new>
  4. Redeploy all Edge Functions to pick up new secret
  5. Audit git history: git log --all -S "service_role" --source --all — if found, use git-filter-repo to purge
  6. Review security_events for any unauthorized DB access during exposure window
  7. Document: exposure start time, end time, which secrets were exposed, what access was possible
CRITICAL — REWARD FARMING

Credit Economy Under Attack

  1. Kill switch: UPDATE feature_flags SET enabled=FALSE WHERE flag_name='rewards_enabled'
  2. Identify suspicious accounts: query credit_ledger for users with >200 credits in 24h
  3. Check for duplicate idempotency_keys or request_hash collisions in processed_requests
  4. Flag all suspicious accounts in fraud_flags
  5. Remove fraudulent credits via negative credit_ledger entries (admin-issued)
  6. Fix the vulnerability (replay, duplicate ID, or logic flaw) in staging first
  7. Re-enable rewards after fix is deployed and verified
CRITICAL — COMPROMISED DEVELOPER MACHINE

Developer Credential Compromise

  1. Immediately rotate ALL credentials that may have been on that machine (all API keys, tokens)
  2. Revoke developer's Supabase access tokens: Supabase Dashboard → Organization → Members
  3. Review security_events for any anomalous actions from that developer's authenticated sessions
  4. Check if production was ever accessed from that machine (it should not be — see Part 11)
  5. Notify developer: advise full machine wipe + credential audit
  6. Review and restrict CI/CD secrets the developer had access to

// supabase/functions/close-trip/index.ts
// Handles: trip close, credit issuance, fresh start
// Security: JWT + step-up token + idempotency + rate limiting + kill switch

import { createClient } from '@supabase/supabase-js'
import { requireAuth, requireStepUp, checkRateLimit, validateTimestamp, withIdempotency } from '../_shared/middleware.ts'

export default async function handler(req: Request) {
  const correlationId = crypto.randomUUID()
  const adminClient = createClient(Deno.env.get('SUPABASE_URL')!, Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!)
  const publicClient = createClient(Deno.env.get('SUPABASE_URL')!, Deno.env.get('SUPABASE_ANON_KEY')!)

  try {
    // 1. Authenticate
    const ctx = await requireAuth(req, publicClient)
    const clientIp = req.headers.get('x-forwarded-for') ?? 'unknown'

    // 2. Rate limit (3 trip closes per hour per user)
    await checkRateLimit(adminClient, ctx.userId, clientIp, 'close-trip', 3, 10, 3600)

    // 3. Parse + validate body
    const body = await req.json()
    const { trip_id, idempotency_key, request_timestamp } = body
    if (!trip_id || !idempotency_key || !request_timestamp) throw badRequestError('Missing required fields')

    // 4. Reject stale requests
    validateTimestamp(request_timestamp)

    // 5. Kill switch check
    const { data: rewardsFlag } = await adminClient.from('feature_flags').select('enabled').eq('flag_name', 'rewards_enabled').single()
    if (!rewardsFlag?.enabled) throw serviceUnavailableError('Rewards are temporarily unavailable')

    // 6. Idempotency: execute once only
    const { result, cached } = await withIdempotency(adminClient, idempotency_key, '/close-trip', ctx.userId, body, async () => {
      // 7. Verify trip belongs to this user (server-side, not trusting client)
      const { data: trip, error } = await adminClient.from('trip_results').select('*').eq('id', trip_id).eq('user_id', ctx.userId).single()
      if (error || !trip) throw forbiddenError('Trip not found or not owned by user')
      if (trip.closed) throw badRequestError('Trip already closed')

      // 8. Run close_week() — atomic Postgres function
      const { data: weekResult, error: closeError } = await adminClient.rpc('close_week', { p_user_id: ctx.userId, p_trip_id: trip_id })
      if (closeError) throw serverError('Failed to close week')

      // 9. Issue credits via secure ledger function (server decides amount)
      await adminClient.rpc('issue_credits', {
        p_user_id:      ctx.userId,
        p_amount:       weekResult.credits_earned,  // server calculated, not client-supplied
        p_type:         'WEEKLY_TRIP',
        p_reference_id: trip_id,
        p_idempotency:  `trip_credit_${trip_id}`,
        p_issued_by:    'system'
      })

      return { success: true, credits_earned: weekResult.credits_earned, new_level: weekResult.new_level, streak: weekResult.streak_days }
    })

    return Response.json({ ...result, cached, correlation_id: correlationId })
  } catch (e) {
    return Response.json({ error: e.message, correlation_id: correlationId }, { status: e.status ?? 500 })
  }
}
