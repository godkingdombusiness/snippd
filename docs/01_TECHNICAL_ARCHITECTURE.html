System Overview & Design Philosophy

Snippd is a mobile-first grocery intelligence platform that combines AI-powered receipt scanning, household budget management, gamified savings tracking, and a curated deal discovery engine. The architecture is designed around three core principles: offline-tolerant UX, household-first data isolation, and zero-trust security.

Design Principles

Household-First
Every data model treats the household as the primary context. Individual users belong to households; carts, spending, and credits are aggregated at the household level.
Zero-Trust Security
RLS policies deny by default. Service role never touches the client. Secrets live in Supabase vault, never in source code or client bundles.
Graceful Degradation
No utility module (audit, security, session guard) ever throws an unhandled exception. All monitoring is fire-and-forget to avoid blocking UI.
Cents-Precision Finance
All monetary values stored as BIGINT cents. No floating-point arithmetic on currency. Formatted at display time only.

State & Data Flow

Snippd uses local component state (useState/useEffect) per screen — no global state management library (no Redux, no Zustand). Data is fetched fresh on focus via useFocusEffect or standard useEffect. This keeps each screen self-contained and avoids stale data bugs.

Pattern: Screen-Local Data Fetching
Every screen that reads from Supabase follows: useState(loading=true) → useEffect(fetchData) → render. Refreshes are triggered by RefreshControl pull-to-refresh or navigation.addListener('focus',...).

Backend Architecture

Supabase Platform

Supabase serves as the complete backend: PostgreSQL database, authentication, row-level security, object storage, and edge function hosting. The client connects via @supabase/supabase-js with a custom SecureStore adapter that stores session tokens in the device's encrypted keychain.

// lib/supabase.js — client initialization
import { createClient } from '@supabase/supabase-js'
import * as SecureStore from 'expo-secure-store'

const ExpoSecureStoreAdapter = {
  getItem:    (key) => SecureStore.getItemAsync(key),
  setItem:    (key, value) => SecureStore.setItemAsync(key, value),
  removeItem: (key) => SecureStore.deleteItemAsync(key),
}

export const supabase = createClient(
  process.env.EXPO_PUBLIC_SUPABASE_URL,
  process.env.EXPO_PUBLIC_SUPABASE_ANON_KEY,
  { auth: { storage: ExpoSecureStoreAdapter, autoRefreshToken: true, persistSession: true } }
)

Google Gemini 1.5 Flash

  • Endpoint: gemini-proxy edge function
  • Input: Base64-encoded receipt image
  • System prompt: Strict grocery item extractor
  • Temperature: 0 (deterministic)

  • Endpoint: gemini-proxy edge function
  • Input: {contents: [{role, parts}]}
  • System prompt: Chef Stash recipe assistant
  • Temperature: 0.7 (creative)
  • Response: Full Gemini candidates structure
  • Used by: ChefStashScreen, freshStart.js

Resend Email API

  • Used exclusively by the security-alert edge function
  • From address: security@getsnippd.com
  • To address: ddavis@getsnippd.com
  • API key stored in Supabase secrets vault (never in client)
  • Emails include: HTML + plain-text fallback, severity color coding, recommended next steps

Supabase Auth (PKCE Flow)

  • Email/password sign-up and sign-in
  • TOTP MFA via Supabase built-in MFA APIs
  • Password reset via magic link (email)
  • Session stored in SecureStore (encrypted device keychain)
  • Auto-refresh: tokens refreshed silently before expiry
  • Session guard: 30-minute inactivity kills session (PanResponder touch tracking)

Section 7

Security Architecture

Security Layers (Defense in Depth)

1
Transport Security — TLS 1.3
All traffic over HTTPS. Expo handles certificate pinning at the native layer. Supabase enforces TLS on all connections.
2
Authentication — Supabase Auth + TOTP MFA
JWT-based sessions stored in SecureStore. TOTP MFA enforced for sensitive operations. Password reset via email magic link.
3
Token Storage — Expo SecureStore (Keychain/Keystore)
Auth tokens never stored in AsyncStorage (plaintext). SecureStore uses iOS Keychain and Android Keystore APIs.
4
Field Encryption — AES-256-GCM
Sensitive fields encrypted client-side before storage via fieldEncryption.js. Key stored in EXPO_PUBLIC_FIELD_ENC_KEY (should be rotated to secure vault).
5
Database — Row-Level Security
Every table has RLS enabled. Users can only read/write their own rows. Admin tables require is_security_admin() helper check.
6
Rate Limiting — Rolling Window Per Endpoint
check_rate_limit() RPC enforces daily limits per user+endpoint. Violations logged to api_rate_limit_log.
7
Security Monitoring — Event Pipeline
SecurityMonitor.js emits events → security-ingest scores/deduplicates → security-alert emails → security-escalate retries/escalates.
8
Auto-Blocking — CRITICAL IP Blocks (24h)
Any CRITICAL severity event auto-blocks the source IP in blocked_indicators for 24 hours. Subsequent requests from that IP are rejected at the Edge Function layer.
9
Audit Trail — Postgres Triggers + App Logging
DML triggers on sensitive tables write to system_audit_logs. App events written via AuditLogger.log(). Logs are append-only (no client DELETE).
